## 二、完整业务流程图（从生产到开锁）

### 阶段 A：设备出生

```
┌────────────────┐
│  生产线工人     │
│  (production_  │
│   operator)    │
└───────┬────────┘
        │ 扫 QR + BLE 采集 MAC
        ▼
┌────────────────┐
│  设备入库       │  status: in_warehouse
│  （厂商仓）     │  owner: vendor
└───────┬────────┘
        │ 厂商管理员操作
        ▼
┌────────────────┐
│  发货到公司     │  status: shipped
│  （填发货单号）  │  owner: 公司A
└───────┬────────┘

        ▼
┌────────────────┐
│  确认入库       │  status: delivered

└───────┬────────┘
        │
        ▼  进入"公司设备总列表"
```

### 阶段 C：日常开锁

```
┌────────────────┐
│   成员张三      │  APP 登录（手机号+密码）
│   (member)     │
└───────┬────────┘
        │ 靠近设备，APP 发起开锁
        ▼
┌──────────────────────────────────┐
│        APP → 后端 API             │
│  POST /api/v1/devices/:id/unlock │
│  携带：用户 JWT token             │
└───────┬──────────────────────────┘
        │ 后端校验逻辑
        ▼
┌──────────────────────────────────────────────────────┐
│              开锁权限校验（5 道关卡）                   │
│                                                       │
│  1. 设备存在？状态是 active？                          │
│  2. 用户属于设备归属公司的成员？                        │
│  3. 查 device_assignment：用户有这条授权记录吗？        │
│     - scope=team：用户属于这个班组吗？                  │
│     - scope=user：用户 ID 匹配吗？                      │
│  4. 授权在有效期内？（validFrom ≤ now ≤ validUntil）     │
│  5. 授权未被撤销？（revokedAt == null）                 │
│                                                       │
│  ALL PASS → 允许开锁，下发 unlock 指令                  │
│  ANY FAIL → 拒绝，「无权限」                            │
└─────────────────────┬────────────────────────────────┘
                      │
          ┌───────────┴───────────┐
          ▼                       ▼
    ┌──────────┐           ┌──────────┐
    │ 校验通过  │           │ 校验失败  │
    │ 下发开锁  │           │ 返回「无权 │
    │ 指令     │           │ 限开此锁」 │
    └──────────┘           └──────────┘
```

## 四、关键业务规则

| # | 规则 | 说明 |
|---|---|---|
| 1 | **设备归属公司** | 设备发货后属于公司，只有该公司的人能操作 |
| 2 | **班组可见性** | 每个班组只能看到分配给本班组的设备 |
| 3 | **人员开锁权限** | 人员通过 APP 只能开自己有授权的锁 |
| 4 | **一张锁一个授权** | 同一时刻一把锁只有一条有效 assignment（旧授权自动撤销） |
| 5 | **分配两级** | 分配到班组 + 可选精细到人员 |
| 6 | **时间窗口** | 授权可以设有效期，到期自动失效 |
| 7 | **授权可撤销** | 管理员随时可以撤掉某人的授权 |
| 8 | **成员移除降级** | 把人从班组移出时，他个人授权降到班组范围（不丢失数据） |

## 五、与现有系统的差距

| 功能 | 当前状态 | 目标 |
|---|---|---|
| 公司详情—设备列表 | ❌ 没有，只有 `deviceCount` 数字 | 完整的设备表格+筛选+操作 |
| 公司详情—设备分配 | ❌ 没有，要去 /devices 全局操作 | 在公司页直接分配 |
| 独立的授权管理页 | ❌ 不存在 | `/authorizations` 页面 |
| 手动撤销授权 | ❌ 没有 UI 入口 | 授权列表里加撤销按钮 |
| 开锁权限校验 API | ❌ 没有 | `POST /api/v1/devices/:id/unlock` |
| 班组设备可见性 | ❌ 没有按班组过滤 | 班组视图只显示自己的设备 |
| 授权计数（maxUses） | ❌ 没有 | 分配时可选 |
| 授权到期处理 | ❌ 没有 | 定时任务检查 |
