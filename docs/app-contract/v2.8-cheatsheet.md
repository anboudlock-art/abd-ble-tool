# APP 契约速查（v2.8 batch 1）

> 给 APP 会话用的权威参考，所有 wire 字段的真实形状。
> 所有源码在 `claude/smart-lock-web-platform-SUvdF` 分支。
> Base URL: `http://120.77.218.138/api/v1`（部署）/ `http://localhost:3001/api/v1`（开发）

## 0. 通用约定

- 所有请求需 `Authorization: Bearer <accessToken>`，登录拿 token。
- BigInt 字段在 wire 上**全部以字符串返回**（前端不会丢精度）：`deviceId: "5"`、`commandId: "123"`。
- 时间戳一律 ISO 8601 UTC 字符串：`"2026-05-02T08:30:12.000Z"`。
- 错误体：`{ code, message, details? }`，HTTP status 表语义（400/403/404/409/500）。

## 2. 我的设备（最重要）

### GET `/users/me/devices`
返回当前用户**有权操作**的设备列表（unioning user-scoped 和 team-scoped，按 validFrom/validUntil 过滤掉过期）。
```jsonc
{
  "items": [
    {
      "deviceId": "5",
      "lockId": "60500001",
      "bleMac": "AA:BB:CC:DD:EE:FF",
      "status": "active",
      "lastState": "closed",        // opened | closed | tampered | unknown
      "lastBattery": 87,
      "lastSeenAt": "2026-05-02T08:00:00.000Z",
      "doorLabel": "2号门",

      // v2.8 加的能力字段 — 决定 APP 渲染哪些按钮
      "model": {
        "id": "3",
        "code": "PADLOCK-V28",
        "name": "4G+BLE+LoRa 挂锁",
        "category": "fourg_padlock",   // gps_lock | eseal | fourg_eseal | fourg_padlock
        "hasBle": true,
        "has4g": true,
        "hasGps": true,
        "hasLora": true,
        "capabilitiesJson": null
      },
      "gatewayId": "1",
      "gatewayOnline": true,         // 远程开锁前 APP 检查这个

      "assignmentScope": "user",     // company | team | user — user-scope 优先
      "teamId": "2",
      "teamName": "一组",
      "validUntil": "2026-12-31T00:00:00.000Z"  // null = 永久
    }
  ]
}
```

**渲染按钮规则**：
- `model.hasBle === true` → 显示「BLE 开锁」按钮（用 link='ble' 走 precheck）
- `model.has4g === true && gatewayOnline === true` → 显示「远程开锁」按钮（用 link='auto' 或 'fourg'）
- `category === 'eseal'` → 只显示 BLE 按钮（电子铅封无远程链路）

## 3. BLE 开锁完整链路（在线）

### Step 1：服务器握手
### POST `/devices/:id/commands`
```jsonc
// req
{
  "commandType": "unlock",        // unlock | lock | query_status
  "link": "ble",                  // 默认 'auto'，BLE 强制传 'ble'
  "phoneLat": 23.1273,            // 可选，请求时位置
  "phoneLng": 113.3528,
  "phoneAccuracyM": 5,
  "occurredAt": null              // 在线时不传；离线补登时传本地时间
}
// resp 201
{
  "commandId": "456",
  "deviceId": "5",
  "lockId": "60500001",
  "bleMac": "AA:BB:CC:DD:EE:FF",
  "link": "ble",
  "expectedCmdId": 200,           // 1 byte (1-255), APP 必须用这个填 BLE 帧的 cmdId 字段
  "timeoutAt": "2026-05-02T08:01:30.000Z",  // 60s 后没 ack 会被 worker 翻 timeout
  "validUntil": "2026-12-31T00:00:00.000Z",
  "serverNote": null              // clamp 时会填字符串
}
```

**5 道权限校验**（任一失败 → 403/405/409，APP 不需要重复实现）：
1. 设备存在且未删
2. 设备 ownerCompany 在调用者 scope 内
3. 设备状态 ∈ {assigned, active}
4. model.hasBle === true
5. 非 admin 调用者必须有非过期的 device_assignment 行

可能的错误：
| HTTP | code | 含义 |
|---|---|---|
| 403 | FORBIDDEN | 跨公司 / 没授权 |
| 404 | NOT_FOUND | 设备不存在 |
| 405 | DEVICE_FEATURE_UNSUPPORTED | model 没 BLE / 选了 lora 但 model 没 lora |
| 409 | CONFLICT | 设备状态不对 / occurredAt 超未来 |

### Step 2：APP 用 expectedCmdId 加密 BLE 帧

```kotlin
// 伪代码（参照 packages/proto/src/ble/index.ts 的 Kotlin port）
val mac = parseMac(bleMac)
val key1 = deriveKey1(mac)
val key2 = deriveKey2(key1, now())   // 当前时间生成
val raw = buildOpenLock(sleepMode = 1, cmdId = expectedCmdId)  // ← 用服务器返回的
val encrypted = encryptRequest(key2, raw)  // 16 byte AES-128-ECB
gattWriteChar.writeValue(encrypted)
```

注意：
- 帧里的 `cmdId` 字段（第 2 byte）**必须用服务器给的 expectedCmdId**
- key 派生：unlock/lock 用 key2（含时间戳），auth 用 key1
- 帧格式：`0x55 cmdId cmd ...params checksum`，pack 进 `[0xFB][len][raw][0xFC pad to 16B]`，再 AES 加密

### Step 3：解密锁回包，回写 ack
```kotlin
val replyBytes = notifyChar.value             // 16 byte encrypted
val raw = decryptResponse(key2, replyBytes)
val resp = parseResponse(raw)                 // { cmdId, cmd, payload }
require(resp.cmdId == expectedCmdId)          // 配对凭据
val ok = resp.payload[0] == 0x00
```

### POST `/device-commands/:id/ack`
```jsonc
// req
{
  "ok": true,
  "errorMessage": null,           // 失败时填
  "occurredAt": "2026-05-02T08:00:30.000Z",  // BLE 写完瞬间的本地时间，必填
  "phoneLat": 23.1273,            // 可选 — 实际开锁地点 GPS
  "phoneLng": 113.3528,
  "phoneAccuracyM": 5
}
// resp 200
{
  "commandId": "456",
  "status": "acked",              // 失败时 "failed"
  "ackedAt": "2026-05-02T08:00:30.500Z",
  "occurredAt": "2026-05-02T08:00:30.000Z",
  "serverNote": null              // 如果服务器 clamp 过会填
}
```

服务器收到后：
1. 翻 status 到 `acked`（ok=true）或 `failed`（ok=false）
2. ok=true 时**自动写一条 lock_event**（source='ble'，eventType=opened/closed），并更新 device.lastState/lastSeenAt
3. ok=false 不写 event，device.lastState 保持原样

## 4. occurredAt clamp 规则（重要）

这条规则同时管 commands 创建 + ack 两个端点：

| occurredAt 值 | 服务端行为 |
|---|---|
| `> now() + 60s` | **400 Bad Request**，APP 必须先校时再重试 |
| `[now() - 7d, now() + 60s]` | 接受为准 |
| `< now() - 7d` | 接受请求，但写入 `now()`，并在 `serverNote` 字段标注 |

APP 端**不要做本地 clamp**，按手机本地时间上传，服务端裁决。

## 5. 离线补登（弱网/隧道）

```
设备已 BLE 连接 + APP 验证有有效 grant（缓存里）
  ↓
APP 自增 localCmdId（不联网）
APP 用 deriveKey1+2 + buildOpenLock(localCmdId) → BLE write
锁回 ACK → 入 pending_commands(syncStatus=pending, serverCommandId=null)

[联网后]
POST /devices/:id/commands
  body: { commandType:"unlock", link:"ble",
          occurredAt: createdLocalAt,
          phoneLat/Lng }
→ 拿到 commandId
POST /device-commands/:id/ack
  body: { ok, occurredAt: executedLocalAt, phoneLat/Lng }
→ syncStatus=acked
```

补登规则：
- 7 天内的 occurredAt 服务端原样接受
- 超 7 天会被 clamp 到 now() 并打 serverNote — 不影响数据落地
- 顺序回放（不要并发），保证 commandId 单调

## 6. 部署照片上传

### POST `/uploads`
```bash
curl -X POST http://120.77.218.138/api/v1/uploads \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/path/to/photo.jpg"
```
```jsonc
// resp 201
{
  "url": "/uploads/2026/05/02/a1b2c3d4-....jpg",  // 用 base + url 拼完整地址
  "sizeBytes": 245678,
  "mimeType": "image/jpeg",
  "originalName": "photo.jpg"
}
```
限制：
- 单文件 ≤ 5 MB
- MIME 必须是 image/{jpeg,jpg,png,webp,heic}
- 单 form 一次只接受一个 file

获取回来：直接 `GET /uploads/2026/05/02/...`，无需 token（图床走 Fastify static）。

部署 API 用：在 POST `/devices/:id/deploy` 的 body 里把这些 url 放进 `photoUrls: ["/uploads/...", ...]`。
