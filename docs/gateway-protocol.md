# 网关 ↔ 平台 通讯协议（Gateway ↔ Platform）

**版本**: v0.1 草案
**适用**: 亿百特 CDEBYTE E90-DTU 系列 LoRa 网关（4G/ETH/NB 回传透传模式）
**面向**: 平台后端实现 + 网关现场配置人员

---

## 0. 设计原则

- 网关是**透传**设备：本身不解析业务，只负责把 LoRa 帧塞进 TCP，把 TCP 字节流塞回 LoRa
- 协议设计要兼容 CDEBYTE DTU 的"注册包 / 心跳包 / 透传数据"三段式行为
- 平台侧把 TCP 字节流切成三类：**注册包 / 心跳包 / 数据帧**
- 任何 DTU 只要按本文档配置注册包和心跳包，就能接入

---

## 1. 传输层

| 项 | 值 |
|---|---|
| 传输协议 | TCP（不是 UDP） |
| 平台端口 | `8901`（生产）/ `18901`（测试） |
| 平台地址 | 由阿里云 SLB 提供固定公网 IP，写入网关配置 |
| 网关角色 | TCP Client，主动连接 |
| 平台角色 | TCP Server，监听并接收多个网关连接 |
| 连接保持 | 长连接，断线由网关自动重连（CDEBYTE 默认行为） |
| 心跳间隔 | 30 秒（网关侧配置）|
| 平台侧超时 | 90 秒未收任何字节 → 服务器主动 close |

---

## 2. 帧格式（统一外层封装）

平台侧从 TCP 字节流中按以下规则切帧。**所有帧都以魔数 `0xAB 0xCD` 开头**：

```
+------+------+------+------+--------------------+
| 0xAB | 0xCD | TYPE | LEN  |   PAYLOAD (LEN B)  |
+------+------+------+------+--------------------+
   1B     1B     1B     1B          1..255 B
```

| 字段 | 长度 | 说明 |
|---|---|---|
| MAGIC | 2B | 固定 `0xAB 0xCD`，用于粘包对齐 |
| TYPE | 1B | 帧类型，见 §3 |
| LEN | 1B | PAYLOAD 长度（不含 MAGIC/TYPE/LEN 自己） |
| PAYLOAD | LEN B | 业务数据 |

> **没有 CRC**：CDEBYTE LoRa 模块在 LoRa 物理层已经有 CRC，TCP 本身也有校验，外层不再加。

> **如果 CDEBYTE DTU 配置工具支持 HEX 格式注册/心跳包**（`SL` 系列默认支持），就把下面的字节串直接配进去；如果只支持 ASCII，看 §6 的 ASCII 替代方案。

---

## 3. 帧类型（TYPE）

| TYPE | 名称 | 方向 | 触发 |
|---|---|---|---|
| `0x01` | REGISTER | GW → 平台 | 网关 TCP 连上后立即发，且只发一次 |
| `0x02` | REGISTER_ACK | 平台 → GW | 收到 REGISTER 后回复，告知网关注册结果 |
| `0x10` | HEARTBEAT | GW → 平台 | 每 30s 发一次 |
| `0x11` | HEARTBEAT_ACK | 平台 → GW | 收到心跳后回复（可选，CDEBYTE 通常不要求 ACK） |
| `0x20` | LORA_UPLINK | GW → 平台 | 网关从 LoRa 收到锁的 11B 上报后透传上来 |
| `0x21` | LORA_DOWNLINK | 平台 → GW | 平台下发开/关锁指令，网关把 PAYLOAD 内容透传到 LoRa |
| `0x30` | GW_CONFIG_GET | 平台 → GW | （预留）查询网关配置 |
| `0x31` | GW_CONFIG_SET | 平台 → GW | （预留）远程配置网关 |
| `0xFE` | ERROR | 任意方向 | 协议错误通报 |

---

## 4. PAYLOAD 详解

### 4.1 REGISTER（0x01）— 网关上线注册

```
PAYLOAD (24B):
+----------+--------+----------+
| GW_ID(8) | TS(4)  | TOKEN(12)|
+----------+--------+----------+
```

| 字段 | 长度 | 说明 |
|---|---|---|
| `GW_ID` | 8B | 网关唯一 ID。**ASCII 数字字符串**，左侧补 `0`，比如 `"00000017"` 表示 17 号网关。出厂时由平台分配并烧录到 CDEBYTE 配置 |
| `TS` | 4B | 当前 Unix 时间戳，big-endian。CDEBYTE 不能动态生成，所以这里固定为 `0x00 0x00 0x00 0x00`（平台不强校验） |
| `TOKEN` | 12B | 鉴权令牌。**ASCII**，平台分配。配错则平台拒绝（回 REGISTER_ACK 错误码后断开） |

**完整注册帧示例**（HEX，38 字节）：

```
AB CD 01 18  30 30 30 30 30 30 31 37  00 00 00 00  41 42 43 44 45 46 47 48 49 4A 4B 4C
└MAGIC┘ T  L └─── GW_ID="00000017" ─┘ └── TS ───┘ └────── TOKEN="ABCDEFGHIJKL" ──────┘
```

### 4.2 REGISTER_ACK（0x02）

```
PAYLOAD (1B):
+----------+
| CODE(1)  |
+----------+
```

| CODE | 含义 | 平台后续动作 |
|---|---|---|
| `0x00` | 注册成功 | 维持连接，开始接收 LoRa 数据 |
| `0x01` | GW_ID 不存在 | 等待 1s 后断开 |
| `0x02` | TOKEN 错误 | 等待 1s 后断开 |
| `0x03` | 网关被禁用 | 立即断开 |

### 4.3 HEARTBEAT（0x10）/ HEARTBEAT_ACK（0x11）

```
PAYLOAD (0B): 空
```

完整帧（4 字节）：`AB CD 10 00`
平台 ACK（4 字节）：`AB CD 11 00`

> CDEBYTE 配置工具里**心跳包内容直接填 `AB CD 10 00`**（HEX 模式）。

### 4.4 LORA_UPLINK（0x20）— 锁上报透传

```
PAYLOAD = LoRa 原始 11B 数据（见 LORA_BLE_Lock_Integration_Guide.md §2.1）

+--------------+---------+---------+--------+----------+
| E220_ADDR(2) | CHN(1)  | MAC(6)  | STAT(1)| BAT(1)   |
+--------------+---------+---------+--------+----------+
```

| 字段 | 长度 | 说明 |
|---|---|---|
| `E220_ADDR` | 2B | 锁的 E220 模组地址（big-endian） |
| `CHN` | 1B | 锁的 LoRa 信道 |
| `MAC` | 6B | 锁的 BLE MAC（用作主键） |
| `STAT` | 1B | `0x01`=开锁 / `0x10`=关锁 / `0x11`=剪断报警 |
| `BAT` | 1B | 电量 0–100 |

完整帧（15 字节）：`AB CD 20 0B  00 08 06  E1 6A 9C F1 F8 7E  10 64`

平台收到后：
1. 按 MAC 在 device 表查锁号
2. 写 `lock_event` 表（事件类型 = stat 映射）
3. 更新 `device.last_state`、`device.battery`、`device.last_seen_at`
4. 推送给订阅者（WebSocket / 推送 / Webhook）

### 4.5 LORA_DOWNLINK（0x21）— 平台下发指令

```
PAYLOAD = LoRa 下行 10B（见 LORA_BLE_Lock_Integration_Guide.md §2.2）

+--------------+---------+---------+--------+
| E220_ADDR(2) | CHN(1)  | MAC(6)  | CMD(1) |
+--------------+---------+---------+--------+
```

| 字段 | 长度 | 说明 |
|---|---|---|
| `E220_ADDR` | 2B | 目标锁的 E220 地址（big-endian） |
| `CHN` | 1B | 目标锁的信道 |
| `MAC` | 6B | 目标锁 MAC（锁会自校验，防错下发） |
| `CMD` | 1B | `0x01`=开锁 / `0x10`=关锁 |

完整帧（14 字节）：`AB CD 21 0A  00 08 06  E1 6A 9C F1 F8 7E  01`

> 网关收到后**剥掉外层 4 字节**（MAGIC+TYPE+LEN），把 PAYLOAD 直接通过 LoRa 发出去。CDEBYTE DTU 在透传模式下默认就是这么干（不会自己加封装）。

> **没有应用层 ACK**。指令是否成功靠锁后续上报的状态变化来确认。平台维护一个"指令-超时"表，超时（比如 10 秒）未见状态变化则记为失败。

### 4.6 ERROR（0xFE）

```
PAYLOAD (2B):
+------+------+
| CODE | DESC |
+------+------+
```

| CODE | 含义 |
|---|---|
| `0x01` | MAGIC 不正确 |
| `0x02` | TYPE 未知 |
| `0x03` | LEN 不合法 |
| `0x04` | PAYLOAD 校验失败 |
| `0x05` | 收到未注册前的数据帧 |

`DESC` 字节预留扩展，目前固定 `0x00`。

## 5. 状态机（平台侧每条 TCP 连接）

```
                    [TCP Accept]
                          │
                          ▼
                ┌──────── INIT ────────┐
                │ 等首帧（10 秒超时）  │
                └─────────┬────────────┘
                          │ 收到 REGISTER
                          ▼
              ┌──────── AUTH ─────────┐
              │ 校验 GW_ID + TOKEN     │
              └──────┬─────────┬──────┘
                成功 │         │ 失败
                     ▼         ▼
            ┌── ESTABLISHED ──┐ 回 REGISTER_ACK 失败 → 1s 后 close
            │ 处理 LORA_*     │
            │ 处理 HEARTBEAT  │
            └────────┬────────┘
                     │ 90s 无字节 / 解析错误
                     ▼
                  [CLOSE]
```

## 6. CDEBYTE DTU 配置说明（给网关现场配置人员看）

进入 CDEBYTE 配置工具（之前 RAR 里的那个 exe）后，需要配置以下项：

| 配置项 | 值 |
|---|---|
| 工作模式 | TCP Client |
| 服务器 IP | （由平台运维提供）|
| 服务器端口 | `8901` |
| 注册包 | **HEX 模式**，填入按 §4.1 计算好的 38 字节内容 |
| 注册包发送时机 | 连接建立后发送一次 |
| 心跳包 | **HEX 模式**，填入 `AB CD 10 00` |
| 心跳间隔 | 30 秒 |
| 重连间隔 | 5 秒 |
| 最大重连次数 | 不限 |
| LoRa 工作模式 | 透传 / FIXED 模式（按 LoRa 帧结构指南配置）|
| LoRa 速率 | 2.4kbps（与锁端一致）|

**ASCII 替代方案**（如固件版本不支持 HEX 注册包）：

```
注册包: REG:00000017:ABCDEFGHIJKL\n
心跳包: HB\n
```

平台侧解析逻辑会做兼容（先按 MAGIC `AB CD` 切，没匹配上再按 `\n` 切并按 ASCII 关键字识别）。

## 8. 后期扩展点

- **MQTT 适配**：如果有大客户的网关只支持 MQTT，加一层 MQTT bridge 把 topic 数据转换成本协议帧再灌入处理管道
- **加密**：当前协议明文。如果跨公网环境对安全敏感，可在 TCP 层包一层 TLS（CDEBYTE 部分高端 DTU 支持 TLS 客户端）
- **网关 OTA**：CDEBYTE 不支持远程升级，将来如果换成自研网关，预留 `0x40 OTA_*` 类型
- **批量下发**：如果有大批量同时开锁的需求，可以加 `0x22 LORA_DOWNLINK_BATCH`（一帧多个 MAC + CMD）
